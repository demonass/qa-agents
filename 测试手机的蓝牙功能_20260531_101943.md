# 手机蓝牙功能测试计划 (v1.0)
**编制角色:** QA Lead  
**适用产品:** 智能手机蓝牙子系统  
**文档状态:** 评审中  

---

## 1. 测试范围 (Scope)

### 1.1 包含范围 (In-Scope)
- **基础控制功能：** 蓝牙开关、可见性/不可见性、配对请求接受/拒绝、连接/断开/重连、设备列表管理、蓝牙设置界面交互
- **通信协议支持：** 经典蓝牙 (BR/EDR) 与低功耗蓝牙 (BLE) 双模支持
- **核心应用场景：**
  - 蓝牙音频：音乐播放、通话、ANC/透传模式切换、多设备音频切换
  - 文件传输：OBEX/FTP 协议下的双向文件传输、断点续传、大文件稳定性
  - 外设连接：键盘、鼠标、手环/手表、TWS耳机、车载蓝牙、蓝牙信标 (Beacon)
- **性能与功耗：** 连接延迟、传输速率、多设备并发切换、蓝牙常驻状态下的电池消耗 (mA/h)
- **异常与容错：** 信号干扰、距离衰减 (1m/5m/10m/15m)、配对失败重试、快速开关机、系统休眠/唤醒后的蓝牙状态恢复
- **兼容性：** 主流 Android/iOS 系统版本、不同厂商蓝牙外设、老旧蓝牙版本设备 (v4.2/v5.0/v5.3)

### 1.2 不包含范围 (Out-of-Scope)
- Wi-Fi、蜂窝网络、NFC、GPS 等其他无线模块
- 第三方 App 内部蓝牙业务逻辑（仅验证系统层蓝牙服务及标准协议）
- 蓝牙底层加密算法与芯片级射频特性（由硬件/基带团队负责）
- 非标准私有协议外设的深度逆向测试

## 2. 测试策略 (Strategy)

> *本策略严格结合软件测试基础知识中的测试类型、用例设计方法与覆盖率要求制定。*

### 2.1 测试类型分级
| 测试类型 | 测试内容 | 执行团队 | 覆盖率目标 |
|:---|:---|:---|:---|
| **单元测试** | 蓝牙驱动/底层服务 API、状态机跳转、日志上报 | 开发团队 | 代码覆盖率 ≥80%（QA 核查） |
| **集成测试** | 蓝牙模块与系统 UI、音频服务、文件系统、外设驱动的交互 | QA + 开发 | 接口契约 100% 验证 |
| **系统测试** | 端到端功能验证（配对、传输、音频、BLE 扫描等） | QA | 功能覆盖率 ≥95% |
| **回归测试** | 每次版本迭代后执行核心用例集，验证无功能退化 | QA | 核心路径 100% 覆盖 |

### 2.2 测试用例设计方法
- **等价类划分：** 有效/无效设备 MAC 格式、标准/非标准密码、合法/非法传输文件类型
- **边界值分析：** 配对失败重试上限（如 5 次）、传输文件大小（1KB ~ 2GB）、有效连接距离（1m / 10m / 15m 临界点）
- **因果图：** 设备状态（开/关/配对/连接） × 用户操作（点击/长按/系统通知） × 环境条件（移动/干扰/休眠） → 生成组合用例
- **错误推测：** 基于历史缺陷经验设计异常场景，如：快速切换飞行模式、多设备并发请求、系统存储满时传输、蓝牙服务崩溃后自动恢复

### 2.3 覆盖率与质量门禁
- **功能覆盖率：** ≥95%（覆盖所有 PRD 需求项）
- **核心路径覆盖率：** 100%（配对→连接→传输/音频→断开）
- **缺陷收敛标准：** P0/P1 缺陷关闭率 100%，P2 缺陷遗留率 ≤3% 且附业务风险说明

### 2.4 自动化与工具策略
- **UI 自动化：** Appium + ADB 覆盖主流程（配对、断开、设置切换）
- **底层接口自动化：** Python + pytest 调用 `bluetoothctl`/系统蓝牙 API 验证状态机
- **协议与性能：** Wireshark + 蓝牙抓包插件（协议合规性）；功率计 + 自动化脚本（功耗测试）

## 3. 资源需求 (Resources)

### 3.1 人力资源
| 角色 | 人数 | 职责 |
|:---|:---|:---|
| QA Lead | 1 | 计划统筹、策略制定、风险管控、质量门禁把控 |
| 高级测试工程师 | 2 | 自动化框架搭建、底层协议验证、性能/功耗测试 |
| 测试工程师 | 3 | 功能测试、兼容性测试、用例执行与缺陷跟踪 |
| 系统/开发支持 | 1 | 日志抓取、底层问题定位、兼容性矩阵维护 |

### 3.2 硬件资源
- **测试终端：** 5 台真机（覆盖不同 SoC 平台与 OS 版本）
- **蓝牙外设库：** ≥15 款（TWS 耳机、音箱、键盘、鼠标、手环、车载模块、信标等，含主流/小众/老旧设备）
- **测试设备：** 射频屏蔽箱、蓝牙协议分析仪（如 Ellisys）、数字功率计、多设备并发切换台

### 3.3 软件与工具资源
- **缺陷管理：** Jira / 禅道
- **自动化框架：** Appium + Python + ADB + Jenkins CI
- **抓包与分析：** Wireshark、`btmon`、`hcitool`/`bluetoothctl`、系统内核日志抓取脚本
- **环境配置：** 标准化测试环境基线（固定 OS 版本、关闭无关后台服务、统一时区与网络）

## 4. 进度安排 (Schedule)

采用敏捷迭代模式，总周期 **6 周**，关键里程碑如下：

| 阶段 | 时间 | 主要活动 | 交付物 |
|:---|:---|:---|:---|
| **Phase 1** | 第 1 周 | 需求冻结、测试计划评审、用例设计（等价类/边界值/因果图） | 《测试用例集 v1.0》《兼容性矩阵》 |
| **Phase 2** | 第 2 周 | 自动化脚本开发、测试环境搭建、冒烟测试执行 | 自动化脚本库、环境就绪确认单 |
| **Phase 3** | 第 3-4 周 | 全量功能测试、兼容性测试、异常场景验证、缺陷修复与复测 | 每日缺陷看板、阶段测试报告 |
| **Phase 4** | 第 5 周 | 稳定性压测（72h 循环）、功耗测试、信号干扰测试、回归测试 | 性能/功耗测试报告、回归结论 |
| **Phase 5** | 第 6 周 | 缺陷闭环验证、测试总结、发布决策 (Go/No-Go) | 《最终测试报告》《发布风险评估》 |

- **例会机制：** 每日站会 (15min) 同步进度与阻塞；每周五缺陷复盘会
- **缓冲策略：** 预留 10% 时间应对需求微调或底层问题定位延迟

## 5. 风险评估与管理 (Risks)

| 风险描述 | 影响 | 概率 | 应对措施 |
|:---|:---|:---|:---|
| **第三方外设兼容性不可控** | 高 | 中 | 建立 Top 20 设备兼容性清单；对不兼容设备输出“已知限制”声明；推动固件升级或提供降级兼容策略 |
| **射频干扰导致测试不稳定** | 高 | 中 | 核心用例均在屏蔽箱内执行；使用频谱仪监控环境；脚本增加重试与超时保护机制 |
| **功耗测试耗时高、人工成本高** | 中 | 高 | 采用自动化功耗采集工具；多机并行测试；脚本化控制设备开关与数据上报 |
| **底层蓝牙驱动缺陷难以定位** | 高 | 低 | 提前准备标准化日志抓取包（含内核/蓝牙服务/应用层）；建立开发-测试联合 Debug 通道；必要时引入芯片原厂技术支持 |
| **测试进度受需求变更影响** | 中 | 中 | 实施用例优先级分级（P0/P1/P2）；采用可插拔测试模块；每日同步进度，预留 10% 缓冲时间 |
